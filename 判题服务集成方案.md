# AIOJ-main 与 remote_judge 判题服务集成方案

## 背景

- **AIOJ-main**：OJ 平台后端（Go 1.21, Gin, MySQL, RabbitMQ），负责用户系统、题目管理、前端展示
- **remote_judge**：独立判题服务（Go 1.25, Docker CLI, gRPC），负责代码编译、沙箱运行、结果判定

### 系统现状

```
AIOJ-main(:8080)                         remote_judge
┌──────────────────────────┐             ┌──────────────┐
│ HTTP API (Gin)           │     gRPC   │ cmd/judger    │
│ 用户系统 / 题目管理       │ ─────────→ │   :9090       │
│ Worker (RabbitMQ)        │  测试用例   │  ← 纯判题引擎  │
│ RemoteJudger (adapter)   │  随请求传输  │  ← 无数据库    │
│ 查 MySQL 获取测试用例      │             │  ← 无队列      │
│ 写入提交结果              │             │  ← 无 Worker  │
└──────────────────────────┘             └──────────────┘
```

**当前集成方式**：AIOJ-main 的 Worker 通过 RabbitMQ 消费提交 → 查 MySQL 拿到测试用例 → gRPC 调用 remote_judge `cmd/judger` → 返回判题结果 → 写回 MySQL。测试用例通过 gRPC 随请求传输，remote_judge 不需要访问任何数据库。

> **关键设计**: `remote_judge` 提供两个入口：
> - `cmd/judger`: 纯判题引擎（gRPC 端口，无队列、无数据库）
> - `cmd/server`: 完整判题后端（HTTP 端口 + 队列 + Worker + 内部 Judger）

---

## 方式一：OJ 后端做薄代理 + remote_judge server

### 架构

```
用户提交代码
  ↓
AIOJ-main（薄代理层）
  ├─ 校验参数（语言、代码长度）
  ├─ 写入提交记录 (status: Pending)
  ├─ HTTP POST 转发到 remote_judge server /api/submissions
  │    只传: submission_id, problem_id, language, code
  │    不传: 测试用例（remote_judge 共享数据库自己查）
  │
remote_judge cmd/server（完整判题后端）
  ├─ 查共享 MySQL 获取测试用例
  ├─ 入队 → Worker → Compile → Run → Compare
  ├─ 返回判题结果
  │
AIOJ-main ← 拿到结果, 更新提交记录
```

### remote_judge 使用的入口

`cmd/server`——完整判题后端（HTTP + 队列 + Worker + Judger + Sandbox）

### 通信方式

HTTP REST API（`POST /api/submissions`, `GET /api/submissions/:id`）

### 题库

共用同一个 MySQL，AIOJ-main 写入题目时 remote_judge 即可读取

### AIOJ-main 需要做的改动

| 操作 | 说明 |
|------|------|
| **去掉** | RabbitMQ Broker、Worker、`RemoteJudger` 适配器 |
| **新增** | HTTP 客户端，向 remote_judge server 转发判题请求 |
| **新增** | 轮询或回调机制，从 remote_judge 获取判题结果并写回数据库 |
| **保留** | HTTP API（接收前端提交）、校验逻辑、数据库写提交记录 |

### remote_judge 需要做的改动

| 操作 | 说明 |
|------|------|
| **修改** | `cmd/server` 配置，连接共享 MySQL |
| **修改** | 或者修改 `POST /api/submissions` 接口，支持题号查询共享库 + 直接传测试用例两种模式 |

### 优缺点

| 优点 | 缺点 |
|------|------|
| 判题队列、Worker 全部收拢到 remote_judge | 数据库耦合：两个项目共用一个 MySQL |
| OJ 后端逻辑简化（去掉整个判题调度） | remote_judge 需要连接 AIOJ 的库查题目 |
| remote_judge 有完整的判题生命周期（队列状态、提交追踪） | OJ 后端改动大，需要去掉现有队列链路 |
| AIOJ 前端可以直接对接 remote_judge（不走 OJ 后端） | HTTP 轮询不如 gRPC 高效 |

## 方式二：gRPC 直调 judger（当前已实现）

### 架构

```
用户提交代码
  ↓
AIOJ-main
  ├─ HTTP API → 入队 RabbitMQ
  ├─ Worker 消费 → 查 MySQL 获取测试用例
  ├─ RemoteJudger 适配器（语言翻译 + 格式转换）
  │   gRPC 调用 remote_judge cmd/judger
  │   测试用例随请求传输
  │
remote_judge cmd/judger
  ├─ gRPC Server → Judger Service → Docker Sandbox
  ├─ 编译 → 运行 → 比对
  ├─ 返回判题结果
  │
AIOJ-main Worker ← 结果写回 MySQL
```

### remote_judge 使用的入口

`cmd/judger`——纯判题引擎（gRPC 端口，无队列，无数据库，无 Worker）

### 通信方式

gRPC（自定义 JSON Codec），方法路径 `/judger.Judger/Judge`

### 题库

不需要共享数据库。AIOJ-main Workter 查自己 MySQL，测试用例通过 gRPC 请求直接传给 remote_judge

```go
// gRPC 请求中包含测试用例
req := &JudgeRequest{
    SubmissionID:  100002,
    ProblemID:     1001,
    Language:      "cpp",
    Code:          "...",
    TimeLimitMS:   1000,
    MemoryLimitMB: 256,
    TestCases: []TestCase{
        {Input: "4 9\n2 7 11 15\n", Expected: "0 1"},
        {Input: "3 6\n3 2 4\n",     Expected: "1 2"},
    },
}
```

### AIOJ-main 已完成的改动

| 文件 | 改动 |
|------|------|
| `internal/judger/adapter.go` | **新建** — RemoteJudger 适配器（语言翻译 + gRPC 调用 + 格式转换） |
| `internal/judger/types.go` | 新增 `JudgerClient` 接口 |
| `internal/mq/worker.go` | `Judger` 字段改为使用 `JudgerClient` 接口 |
| `cmd/server/main.go` | 根据 `config.remote` 选择 Client / RemoteJudger |
| `cmd/judger/main.go` | 新增 `--remote` 标志，支持代理模式 |
| `internal/config/config.go` | `JudgerConfig` 新增 `Remote bool` 字段 |
| `config.yaml` | `remote: true`，指向 `127.0.0.1:9090` |

### remote_judge 改动

**无**。`cmd/judger` 不需要任何改动，它在 gRPC 上接收请求、判题、返回结果。

### 优缺点

| 优点 | 缺点 |
|------|------|
| 完全解耦：remote_judge 不依赖任何外部数据库 | OJ 后端需要维护队列和 Worker |
| gRPC 高效通信（单次往返 < 1ms） | 测试用例数据通过 gRPC 传输（数据量极小，实际开销可忽略） |
| remote_judge 可独立扩容（多实例无状态） | AIOJ 后端代码量多于方式一 |
| OJ 后端改动小（当前已完成） | |
| 支持 mock / real sandbox 一键切换（`remote: false`） | |
| remote_judge 的 `cmd/server` 保留其他用途 | |

## 两种方式对比总结

| 维度 | 方式一（代理 + server） | 方式二（gRPC + judger） |
|------|----------------------|----------------------|
| remote_judge 入口 | `cmd/server` | `cmd/judger` |
| 通信协议 | HTTP | gRPC（JSON Codec） |
| 题库 | 共享 MySQL | 测试用例 gRPC 传输 |
| 数据库耦合 | 是 | 否 |
| 判题队列归属 | remote_judge | AIOJ-main |
| OJ 后端职责 | 薄代理 | 完整调度 |
| AIOJ 后端改动量 | 大（去队列+Worker+适配器，加转发了逻辑） | 小（已完成） |
| remote_judge 改动量 | 需要（连接共享库或改造接口） | 无 |
| 服务可扩容性 | 一般（都依赖同一个数据库） | 高（remote_judge 无状态） |
| 当前状态 | **未实施** | **已实施并验证通过** |

---

## 推荐

**当前项目使用方式二**，原因：

1. **完全解耦** — remote_judge 不依赖 AIOJ 的数据库，可独立扩容
2. **改动量最小** — AIOJ 和 remote_judge 都是最小化改动
3. **已有 gRPC 基础** — 双方 proto 一致、codec 一致、已验证通过
4. **remote_judge 可独立演进** — `cmd/server` 保留给需要完整判题后端的场景，`cmd/judger` 给集成场景
5. **服务可替换** — 通过配置 `remote: false` 可瞬间切回 MockSandbox

如果未来需要方式一（例如多个 OJ 平台共享一个 remote_judge 实例、由 remote_judge 统一管理队列），可以在此基础上增加 HTTP 转发层，无需废弃现有 gRPC 集成。
